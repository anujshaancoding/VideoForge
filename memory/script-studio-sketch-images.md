CEO wants Script Studio to turn a narration/story script into a 4-5 min narrated **sketch video**: per-scene pencil/pen/color-sketch images synced to AI voiceover. Hard constraint: **ZERO recurring spend — local or genuinely-free only** (no paid API, no card). CEO researched the dev community expecting a free path; we found one.

**Chosen architecture (decided 2026-06-16, CEO):** decouple base-image generation from sketch styling.
- **Base image:** Draw Things (free Mac app, Metal, A1111-compatible HTTP `/sdapi/v1/txt2img`, seed-controllable, no key/limit) on the host M4/16GB — ~8-15s/image. Keyless **Pollinations.ai** is the cloud fallback (rate-limited ~1 img/15s; returns HTTP 200 + placeholder on throttle, so validate outputs).
- **Sketch styling + consistency:** a pure deterministic **OpenCV filter** applied to every image — 3 styles: `graphite` (dodge+Laplacian+contrast), `pen` (adaptiveThreshold line art), `color` (`cv2.pencilSketch`). The uniform filter is what makes 30-40 images look consistent despite model drift; it's golden-testable (fits the WYCIWYG invariant culture).

**Why this fits:** Script Studio v2 already does script→scene-plan→TTS(WAV per scene, $0)→`assemblePlannedProject`→export, with sync solved by probed VO duration as source of truth. The only new piece is generating sketch images into the existing **Arrange** b-roll slot (per-scene photo asset spanning the VO window). See [[script-studio-proof-harness]].

**MINIMAL LINE-ART look + FLUX-first (2026-06-27, CEO):** CEO's reference = delicate single-continuous-line ink art (thin black line on near-white, generous negative space, almost no shading), deliberately MINIMAL so the narration + captions stay the hero. Key finding: the old approach (render a *detailed* image then threshold it with the `pen` filter) FOUGHT this — thresholding a busy render = muddy crosshatch, and even on a clean base the pen filter over-cooks it to uniform solid black, losing the airy delicacy. Fix shipped in apps/api/src/script:
- **Prompt FOR line art** (sketchScenes.ts `promptFor`, now style-aware): sketch styles get `single continuous line drawing of <subject>, minimalist fine-line ink illustration, delicate thin black linework, clean white background, generous negative space, one subject, elegant and simple, line art`. Photo mode keeps the old detailed prompt. (On SDXL-Turbo CFG≈1 the negative prompt is inert, so the POSITIVE prompt is the only lever — added a richer negativePrompt anyway for FLUX/engines that honor it.)
- **New `line` IllustrationStyle = the default** (sketch.ts: DEFAULT_ILLUSTRATION_STYLE='line'; web SKETCH_STYLE_OPTIONS + DEFAULT_SKETCH_STYLE='line'). 'line' keeps the AI base UNFILTERED — finishFrame cover-fits it (no applySketch), because the base is already line art. pen/graphite/color still available for the filtered looks.
- **Engine: FLUX/Pollinations FIRST** (imagegen.ts: 'auto' now tries Pollinations before Draw Things). FLUX renders the reference aesthetic far better than local SDXL-Turbo; CEO accepted the tradeoff (cloud dep, prompts leave the machine, ~1 img/15s) over local DT. DT stays as offline fallback / primary when IMAGEGEN_ENGINE=drawthings. Bonus: FLUX is Apache-licensed = commercial-safe (vs SDXL-Turbo non-commercial). Verified: FLUX base for woman/cat/house matched the reference pins beautifully (samples were in /tmp/vf-linestyle). api+web typecheck clean; no script tests broke (the 2 failing apps/api watermark contract tests are the known pre-existing baseline failure — see line above).

🐞 **WORKER BUG FOUND + FIXED via E2E (2026-06-28):** after the worker rebuild, exports succeeded but every frame showed the FULL caption (no reveal). Root cause: `apps/render-worker/src/worker.ts` wrote each overlay text-file to a path keyed by `${exportId}-${overlayId}` — but a typewriter overlay emits MANY tokens (one per reveal step) sharing ONE overlayId, so they all collided onto one file; the last write (the full-text final step) won, and every per-step `textfile=` token resolved to full text. Fix: key the temp filename on the unique `tf.token` (sanitised), not overlayId. Rebuilt + re-rendered → typewriter now types char-by-char in the real MP4 (verified frames: 0.10s "St" → 0.40s "Start y" → 0.70s "Start your m" → 1.10s "Start your morning"). The ffmpeg-graph unit tests verify the graph + textFiles ARRAY but NOT the worker's consumption of it — that gap is why this slipped to E2E.
